# Linux Kernel Drivers for Car Control

Реализация комплекса низкоуровневых драйверов символьных устройств ядра Linux для управления основными периферийными узлами машинки на базе архитектуры Raspberry Pi 5.

## Архитектура

Система управления разделена на три независимых модуля ядра, обеспечивающих изоляцию контекста устройств, потокобезопасность и асинхронную обработку аппаратных событий:

### 1. LED Controller (`/led_module`)
* **Назначение:** Управление фарами/индикацией автомобиля.
* **Логика работы:** Модуль регистрируется в системе как символьное устройство и создаёт узел устройства `/dev/led_controller`. 
  * При записи (`write`) строка из User Space копируется в память ядра через `copy_from_user`. Встроенный парсер очищает строку от пробелов и переносов строк (`\n`), после чего сопоставляет её с командами `on` или `off`.
  * При обнаружении команды состояние транслируется на физический GPIO-пин с помощью низкоуровневых функций подсистемы GPIO ядра Linux. 
  * Доступ к изменению состояния защищен мьютексом для предотвращения Race Conditions при параллельных запросах.

### 2. Motors Controller (`/motors_module`)
* **Назначение:** Драйвер ходовых двигателей (движение вперед, назад, остановка).
* **Логика работы:** Управляет одновременно 4 GPIO-каналами: два пина на левый мотор, два — на правый.
  * Драйвер принимает высокоуровневые текстовые команды: `forward`, `backward`, `stop`.
  * При обработке команды `forward` или `backward` логика модуля переключает соответствующие пины в комбинацию `HIGH/LOW` для задания направления вращения. При команде `stop` все 4 пина сбрасываются в `LOW`.
  * Вся критическая секция внутри `device_write`: чтение команды, парсинг, изменение состояний пинов обернута в `mutex_lock` и `mutex_unlock`. Это гарантирует, что если один поток выполняет команду `forward`, параллельный запрос `stop` из другого потока не приведет к непредсказуемому переключению транзисторов и короткому замыканию.

### 3. Steering Controller (`/steering_module`)
* **Назначение:** Модуль рулевого управления повороты влево/вправо с аппаратной защитой.
* **Логика работы:** Самый сложный модуль, использующий асинхронную обработку событий реального времени. Помимо управления пинами поворота рулевого электропривода, модуль отслеживает состояние физических концевых выключателей крайних положений руля.
  * **Аппаратные прерывания (ISR):** При инициализации модуль регистрирует обработчики прерываний (`request_irq`) для пинов концевиков, реагирующие на нажатие выключателя.
  * **Разделение обработки:** В соответствии с требованиями разработки драйверов, обработка прерывания разделена.
    * *Top-half:* Функция обработки прерывания выполняется мгновенно, фиксирует факт достижения рулем края и ставит задачу в системную очередь отложенных вызовов ядра (`schedule_work`).
    * *Bottom-half:* Асинхронный обработчик `endstop_work_handler` вызывается ядром в безопасном контексте потока чуть позже. Он захватывает мьютекс управления, мгновенно обесточивает пины мотора руля для предотвращения механической поломки привода и обновляет статус устройства на `STOP`.


## Сборка и развертывание
Так как каждый модуль полностью автономен, сборка и управление выполняются из директории конкретного компонента.

Номера физических GPIO-пинов для подключения периферии объявлены как параметры модуля ядра (module_param_named). Это позволяет гибко менять схему подключения компонентов без повторной перекомпиляции исходного кода.

Передача номеров пинов происходит в момент загрузки скомпилированного .ko файла в ядро с помощью утилиты insmod. Если параметры не переданы, драйвер автоматически инициализирует пины по умолчанию (базовый пин 100 и далее).

Применяемые команды (на примере led_module/):
```bash
cd led_module/

# 1. Стандартная сборка модуля ядра
make

# 2. Загрузка модуля с явным указанием GPIO-пина (например, 518)
sudo insmod led_module.ko gpio_pin=518

# 3. Выгрузка модуля из пространства ядра
sudo rmmod led_module

# 4. Очистка артефактов компиляции
make clean
```
Примеры установки для остальных модулей с параметрами пинов:
```bash
# Для ходовых двигателей (4 пина):
cd ../motors_module/

sudo insmod motors_module.ko left_fwd_pin=519 left_bwd_pin=520 right_fwd_pin=522 right_bwd_pin=523

# Для рулевого управления (2 пина мотора + 2 пина концевых выключателей):
cd ../steering_module/

sudo insmod steering_module.ko left_motor_pin=530 right_motor_pin=531 left_endstop_pin=535 right_endstop_pin=537
```

Вы также можете использовать встроенные в Makefile автоматизированные команды установки make install и удаления make remove, которые уже содержат преднастроенные наборы пинов.

Применяемые команды (на примере led_module/):

```bash
cd led_module/

# 1. Стандартная сборка модуля ядра
make

# 3. Загрузка модуля с предустановленным номером GPIO-пина
make install

# 4. Выгрузка модуля из пространства ядра
make remove

# 5. Очистка артефактов компиляции
make clean
```

Аналогичные команды применяются внутри папок motors_module/ и steering_module/

## Тестирование системы
В проекте реализовано модульное и интеграционное покрытие тестами.

Модульные тесты проверяют изолированную внутреннюю бизнес-логику модулей: парсинг входящих строк от пользователя, очистку от лишних пробелов, обработку управляющих символов \n, корректность граничных условий и форматирование строк вывода статуса.

Для сборки модулей со встроенными модульными тестами перейдите в папку нужного модуля и выполните:
```bash
make test
```

Команда `make test` компилирует модуль с флагом `-DTEST_MODE`,который активирует модульные тесты при загрузке модуля.

Тесты автоматически запускаются в момент загрузки модуля. Результаты тестирования (статусы [OK] или [FAIL]) выводятся в системный лог ядра. Посмотреть результаты можно через утилиту dmesg:
```bash
dmesg | grep -E "ЗАПУСК|ТЕСТ|LED|MOTORS|STEERING"
```

Интеграционные тесты имитируют полный цикл работы операционной системы с устройствами: автоматизируют компиляцию, загрузку модулей с передачей параметров GPIO-пинов, проверяют корректность регистрации нод устройств в каталоге /dev/, отправляют управляющие команды и считывают текущее состояние устройств.

Интеграционные тесты требуют прав суперпользователя (sudo) для загрузки/выгрузки модулей ядра и доступа к устройствам в /dev/.

Запуск тестов интеграции выполняется из папки соответствующего модуля:
```bash
# Тестирование модуля LED
cd led_module/ && sudo chmod +x led_module_integration_test.sh && sudo ./led_module_integration_test.sh

# Тестирование модуля моторов
cd ../motors_module/ && sudo chmod +x motors_module_integration_test.sh && sudo ./motors_module_integration_test.sh

# Тестирование рулевого управления
cd ../steering_module/ && sudo chmod +x steering_module_integration_test.sh && sudo ./steering_module_integration_test.sh
```
